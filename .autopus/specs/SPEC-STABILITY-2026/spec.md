## 1. 배경 / 문제 정의

매일 1~5개 릴리즈가 나가는데 릴리즈마다 회귀가 터져 "수정 → 새 문제 → 수정" 루프에 갇혀 있다.
무료체험자가 첫 발행에서 누락/오류를 보고 이탈한다. 사용자 결정: **기능 동결 + 전면 안정화**.

### 구조적 근본 원인 (왜 "매번" 터지는가)

1. **에러 삼킴 87건**: automation 폴더의 `catch(() => null)` 류 — 실패해도 발행이 계속 진행돼 "반쪽 발행"이 나간다. S1·S3·S5의 공통 뿌리.
2. **발행 직전 최종 검증 부재**: 글에 들어가야 할 요소(구분선/이전글카드/CTA/이미지 수)가 다 들어갔는지 확인하는 게이트가 없다.
3. **중복 구현**: 본문 타이핑·이전글 삽입 로직이 naverBlogAutomation.ts와 editorHelpers.ts 양쪽에 존재 → 한쪽만 고쳐지는 회귀.
4. **테스트 커버리지 3.7% + 정적 텍스트 매칭 테스트의 한계**: "메서드가 존재한다"만 검증하고 "호출된다"는 검증 안 함 (S2가 8개월간 dead code였던 이유).
5. **번들 릴리즈 커밋**: "vX.Y.Z release" 하나에 수십 변경 → bisect 불가, 회귀 원인 추적에 반나절씩 소모.
6. **god file 4개** (naverBlogAutomation 9.7K, contentGenerator 14K, renderer 8.8K, main 8K줄): 수정 반경 예측 불가.

## 2. 목표

- **"오류 0%"가 아니라 "반쪽 발행 0%"**: 외부 의존(네이버 UI, AI provider) 때문에 실패 자체는 0이 될 수 없다. 대신 *실패하면 명확한 사유와 함께 중단하고, 성공하면 완전한 글*이라는 계약을 코드로 강제한다 (Pre-publish Assertion).
- **회귀 0 운영**: 모든 수정에 가드 테스트 선행(RED→GREEN) + 1릴리즈 1~3픽스 + 라이브 스모크 체크리스트 + 단계적 공개.
- 안정화 기간(약 4주) **신규 기능 동결**. 사용자들은 마지막 안정 버전 유지, 안정화 릴리즈는 검증 후 공개.

## 3. 회귀 0 보장 메커니즘 (모든 Phase 공통 게이트)

릴리즈마다 아래 7개를 전부 통과해야 출고:

1. **가드 테스트 선행**: 수정 전 재현/가드 테스트 작성 → RED 확인 → 수정 → GREEN (이번 R1에서 실증: 6 RED → 99 GREEN)
2. **vitest 전체 GREEN** (현재 2,884개+, 1실패도 출고 불가 — 기준선 정상화 완료)
3. **lint 0 errors + build 성공** (exit 0)
4. `npm run self-test` 통과
5. **라이브 스모크 체크리스트** (plan.md §R별 정의): 패키징 후 실제 더블클릭 실행 → 풀오토 1건 + 반자동 1건 발행 → 발행물 검수(구분선·이전글카드·CTA·이미지 순서·썸네일)
6. **단계적 공개**: 본인 계정 1일 운용 → 이상 없으면 다운로드 페이지 공개. 문제 시 즉시 이전 버전 링크 복원(롤백 경로 상시 유지)
7. **릴리즈 커밋에 변경 나열** (bisect 가능하게): `fix: A`, `fix: B`, `chore: vX.Y.Z 버전업` 분리 커밋
