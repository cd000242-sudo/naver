## 2. 의사결정 근거

### 2.1 왜 silent 엔진 폴백 금지인가

**근거 1**: 메모리 `feedback_no_fallback.md`
> 사용자 선택 모델 실패 시 silent 폴백 절대 금지. 차단형 모달 + 사용자 명시 동의로 처리 (v2.7.94, v2.8.3)

**근거 2**: git log 회귀 사례
- `8dc81b39 fix: v2.10.56 silent 폴백 회귀 + 제목 재생성 3회→1회 (사용자 선택 모델 100% 존중)`
- `53f219ae fix: v2.10.57 이미지 프롬프트 변환 멀티모델 자동 폴백 회귀 (사용자 선택 모델만 사용)`

**결론**: 같은 엔진 내 자동 복구는 OK, 다른 엔진 전환은 무조건 차단형 모달.

### 2.2 왜 R5 활성도 폴링 30분 상한인가

**근거**:
- 현재 ImageFX 5분(925줄), Flow 10분(451줄) 고정 — 2FA 푸시·SMS 늦은 사용자 다 잃어버림.
- 무제한 대기는 사용자가 자리 비웠을 때 메모리/리소스 누수.
- 30분이면 통상적인 2FA 우회·계정 전환·SMS 재전송 모두 가능.

### 2.3 왜 R6 5초 후 1회 재요청인가

**관찰**: Flow는 generation 직후 우선 썸네일(<1KB)을 반환하고, ~5초 후 풀해상도로 교체하는 패턴.
**근거**: `flowGenerator.ts:921` 현재 즉시 reject — 풀해상도 교체 전에 버려지는 케이스.
**결정**: 5초는 보수적 추정 — 실제 운영 데이터로 조정 (운영 대시보드 D-3 참고).

### 2.4 왜 R3 같은 세션 내 AdsPower 재시도 안 함

**문제**: 같은 세션 내 재시도하면 이전 시도의 부분 상태(쿠키·페이지)가 섞임 → 디버깅 불가.
**대안**: 다음 앱 재시작 시 다시 시도 — 사용자가 의도적으로 끄지 않는 한 자동 복구.

### 2.5 왜 차단형 모달인가 (vs 알림 토스트)

**근거**: 메모리 `feedback_no_fallback.md` — 차단형 모달 + 사용자 명시 동의로 처리.
**부가 근거**: 토스트는 사용자가 놓치고 다음 헤딩 진행되면 silent 폴백과 같은 효과.
**결론**: 회복 가능 → 토스트 / 회복 불가능 → 모달.

## 3. 외부 조건 분석 (앱이 못 푸는 영역)

### 3.1 한국 IP 차단 (HTTP 403)

**Google 정책**:
- ImageFX는 일부 지역에 베타 한정 — 미국 외 지역 차단 가능.
- VPN 경유 정상 작동 사례 다수 (커뮤니티 보고).

**우회 가능성**:
- 외부 프록시 등록 (이미 UI 있음) — B1 모달에서 안내.
- Google Workspace 계정으로 변경 — 권한 다를 수 있음.
- VPN 자동 활성화는 앱 권한 밖 (사용자 시스템 권한 필요).

### 3.2 2FA / Google Workspace 정책

**제약**:
- 2FA 코드는 사용자만 입력 가능.
- Workspace 관리자가 ImageFX/Flow 접근을 차단했으면 해제 권한 없음.
- 일부 학교/회사 Workspace는 외부 OAuth 차단.

**대응**: B5/B7 모달로 정확한 안내 + 다른 계정 선택 유도.

### 3.3 Flow UI 변경 (Google 측)

**관찰**:
- Flow는 베타 단계 — UI 변경 빈도 높음.
- git log: `a50f778d feat: v1.4.88 Flow UI 자동화 전면 재작성`

**대응**:
- R4 다중 셀렉터: 단기 안전장치
- remoteUpdate 강제 갱신: 1시간 내 패치 배포 시 사용자 자동 적용
- B6 모달: 셀렉터 모두 실패 시 사용자에게 정확히 안내

## 5. 리스크와 대응

### 5.1 자동 재시도가 한도를 더 빨리 소진

**우려**: R1~R6 모두 활성화하면 Google API 호출 횟수 증가 → 429 빨리 도달.
**완화**:
- R-3: 자동 복구 N회 누적 시 모달 격상
- 헤딩 단위 카운터 리셋으로 같은 종류 에러 무한 반복 차단
- 운영 대시보드 D-3에서 자동 복구 횟수 모니터링

### 5.2 다중 셀렉터(R4)의 잘못된 클릭

**우려**: 우선순위 폴백 셀렉터가 다른 버튼 매칭 → 의도치 않은 동작.
**완화**:
- 통합 테스트 시나리오 B에서 잘못된 매칭 검증
- 각 셀렉터에 `data-testid` 우선 (Google이 제공할 시)
- 모니터링: 클릭 후 URL 변화·DOM 변화 검증

### 5.3 차단 모달 피로도

**우려**: 모달이 너무 자주 뜨면 사용자가 무시·앱 신뢰도 저하.
**완화**:
- 운영 대시보드 D-3에서 노출 빈도 모니터링
- 같은 코드 5회 이상 누적 노출 시 메시지 변경 ("이 문제가 반복됩니다 — 다른 환경 시도")
- 사용자가 동일 모달 "취소" 3회 → 다음 노출은 토스트로 격하 검토

## 6. 대안 검토

### 대안 A: 모든 실패를 모달로 처리
- **장점**: 단순.
- **단점**: 일시 timeout 같은 자동 복구 가능 케이스도 사용자 개입 → 피로도 폭증.
- **결정**: 기각.

### 대안 B: 모든 실패를 silent 자동 폴백
- **장점**: 사용자 개입 0.
- **단점**: 메모리 원칙 위배. 디버깅 불가. 사용자가 "왜 안 됐는지" 모름.
- **결정**: 기각.

### 대안 C (채택): 회복 가능/불가능 분류 후 차등 대응
- **회복 가능** → 자동 복구 + 토스트 알림
- **회복 불가능** → 차단형 모달 + 사용자 명시 동의
- **결정**: 채택.
