## 2. 설계 결정

### 2.1 왜 4단계(P0~P3)로 나누는가
- P0만으로도 환각 "생성 압력"은 크게 낮아지지만, 입력 소스가 계속 비면 콘텐츠 품질이 스펙 나열 수준으로 고정됨.
- P1에서 블로그 댓글 소스를 공급해야 진짜 "사용자 목소리" 기반 생성이 가능.
- P2는 P0 변경(조건부 블록)을 전제하므로 뒤에 오며, P3는 P1의 `userVoice` 입력을 활용해야 의미 있는 ground truth 매칭 가능.
- 결과적으로 **입력 차단(P0) → 입력 확충(P1) → 분량 탄력화(P2) → 출력 검증(P3)** 의 환각 방지 풀스택이 형성됨.

### 2.2 P0 접근법: "치환" vs "제거"
- **선택**: 치환(리뷰 블록 → "스펙/공식 설명 기반 분석" 블록)
- **이유**: 제거 시 프롬프트 구조가 비대칭이 되어 LLM이 "누락된 후기"를 상상할 위험. 명시적으로 "스펙 기반"을 지시하면 환각 압력이 낮아짐.
- **대안 검토**: 리뷰 섹션 완전 삭제 → LLM이 내부 지식으로 후기 꾸며낼 위험 존재.

### 2.3 P1 접근법: 기존 `commentCrawler` 재활용 vs 신규 크롤러
- **선택**: 기존 `commentCrawler.ts` export 재정비 + `naverBlogCrawler.ts`에 통합
- **이유**: 중복 크롤링 로직 회피(파일 수 증가 억제, 300줄 제한 준수). 기존 셀렉터/안정화 로직 재활용.
- **대안 검토**: 신규 `blogCommentCrawler.ts` 작성 → 셀렉터 중복, 유지보수 부담.

### 2.4 P2 공식 설계: `baseLen + perReview × N`
- **선택**: `baseLen=800, perReview=200, 상한=2,200, 리뷰0+스펙만=1,200`
- **이유**:
  - SEO 하한은 일반적으로 800~1,000자 → `baseLen=800`이면 충돌 최소화.
  - 리뷰 1건당 200자는 경험 요약에 필요한 최소 단위(기존 발행물 분석 기반 추정).
  - 리뷰 10건 이상이면 2,200자 클램프로 과장 방지.
  - 리뷰 0개 + 스펙만일 때 1,200자 상한은 "분량 부족으로 인한 환각" 완전 차단.

### 2.5 P3 접근법: 규칙 기반 추출 vs LLM 기반 검증
- **선택**: 규칙 기반 추출(정규식 + 토큰 유사도) → 원문 매칭
- **이유**:
  - LLM 기반 검증은 비용 2배, 그 자체가 환각 가능성 보유.
  - 수치/기간/비교는 정규식으로 높은 재현율 확보 가능.
  - 오탐은 1회 재생성 정책으로 완화.
- **대안 검토**: 검증용 LLM 콜 추가 → 비용/지연/이중 환각 리스크로 기각.

### 2.6 Feature Flag 체계
- 각 P단계별 독립 flag: `REVIEW_GUARD_V1`, `USER_VOICE_COLLECT`, `TARGET_LEN_ADAPTIVE`, `FACT_CHECK_V1`
- **이유**: 네이버 DOM 변경·LLM 응답 회귀 등 외부 원인 시 단일 flag OFF로 격리 가능.
- **기각 대안**: 통합 flag `REVIEW_GUARD_ALL` → 부분 롤백 불가.

## 4. LLM 프롬프트 엔지니어링 참고

환각 방지에 관한 공지 기법 중 본 SPEC에 반영된 패턴.

### 4.1 Grounded Generation
- **원칙**: LLM이 출력하는 모든 사실 진술은 제공된 소스 문서(원문)에 근거해야 한다.
- **적용**: P0에서 `reviewAvailable=false`일 때 "스펙/공식 설명 기반 분석"으로 치환하는 것이 grounded generation의 최소 형태.
- **참고 개념**: Retrieval-Augmented Generation(RAG)의 축소판 — 본 SPEC은 외부 검색을 하지 않고, 이미 수집된 스펙/리뷰만을 ground truth로 사용.

### 4.2 Self-Consistency
- **원칙**: 동일 프롬프트에 대해 여러 번 생성 후 일관된 답변만 채택.
- **적용**: P3에서 환각 판정 문장 "1회 재생성" 정책이 약식 self-consistency. 2회 실패 = 일관된 환각 생성 경향으로 판단 → 문장 삭제.

### 4.3 Constraint Injection
- **원칙**: 프롬프트에 "X 금지", "Y만 허용" 같은 경계 조건을 명시.
- **적용**: P0 프롬프트 수정에서 "리뷰 데이터 없을 경우 해당 섹션 생략, '써본/2주 테스트' 표현 금지" 삽입.
- **한계**: Constraint만으로는 완전 차단 어려움 → P3 사후 검증이 필요한 이유.

### 4.4 Length-Decoupled Instruction
- **원칙**: "X자 작성"과 "사실만 작성"은 충돌할 수 있으므로, 분량 지시를 사실 지시 위에 두지 않는다.
- **적용**: P2에서 분량을 입력 데이터량에 연동 → 데이터 부족 시 분량 자동 하향.
