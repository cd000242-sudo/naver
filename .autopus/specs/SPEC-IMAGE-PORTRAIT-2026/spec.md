# SPEC-IMAGE-PORTRAIT-2026: 실존 유명인 이미지 초상권/퍼블리시티권 게이트

## 1. 목적/배경

**결론: 연예/실존인물 글에 자동 삽입되는 사진(크롤링/AI 생성)은 텍스트 명예훼손과 완전히 별개 트랙인 초상권·퍼블리시티권 침해를 일으키며, 텍스트 가드(SPEC-DEFAMATION-2026)로는 원천적으로 커버 불가하다. 상업성(애드포스트·제휴)이 리스크를 급상승시킨다. 본 SPEC은 '실존 유명인 식별 이미지의 자동 삽입 차단 + 라이선스 명확 대체 유도'를 설계한다.**

### 1.1 법적 근거 (legal-research.md Q7 — 검증된 사실)
- **초상권** = 헌법상 인격권, 명문근거 불요·확립 판례. 백지영·남규리 각 500만원(서울중앙지법 2013), 박효신 초상권+명예권 3천만원 확정.
- **퍼블리시티권** = 개정 부정경쟁방지법 제2조 제1호 **(타)목**(2022-06-08 시행): "널리 인식되고 경제적 가치를 가지는 성명·초상 등을 공정 상거래관행에 반해 무단사용" → 손배·금지청구·특허청 시정권고.
- **AI 생성 lookalike도 성립**: '실존인물 닮음'이면 초상권·퍼블리시티권 성립(머니투데이 2026-06-19 이지혜 딥페이크). 성적 딥페이크는 성폭력처벌법 별도.
- **크롤링 사진 = 삼중 침해**: 초상권 + 퍼블리시티권 + **사진저작물 저작권**.
- **핵심 차이**: 초상권/퍼블리시티권은 '거짓 사실'을 요건으로 하지 않는다 → 사진이 사실이고 명예훼손이 없어도 '동의 없는 상업적 이용' 자체로 침해 성립 → **텍스트 팩트체크로 커버 불가**.

### 1.2 이 앱이 위험한 이유
1. 이미지 파이프라인(`src/image/`, `src/crawler/`)이 키워드 기반으로 실존인물 사진을 크롤링·삽입할 수 있다.
2. Dropshot(리더스 나노바나나) 등 AI 이미지 생성이 실존인물 lookalike를 만들 수 있다.
3. 애드포스트·제휴 수익화가 '순수 비영리·정보목적' 방어를 봉쇄한다.
4. 자동 대량발행이라 사람 검토 없이 침해 이미지가 누적된다.

### 1.3 범위 밖 / 잔존 (정식자문 유효)
- (타)목의 AI 생성 유명인 이미지 적용 확정 대법원 판례는 축적 단계(일부 unsettled).
- AI기본법 표시의무의 1차 주체는 AI 서비스 사업자 — 표시 이행이 초상권 침해를 면책하지 않음.
- 순수 비영리·보도/논평 목적 예외는 자동 대량발행 앱에 적용 곤란.

## 2. 범위

### 2.1 대상 이미지 경로
| 경로 | 소스 | 리스크 |
|---|---|---|
| 크롤링 삽입 | `src/crawler/`, `naverBlogCrawler.ts`, 이미지 검색 수집 | 삼중(초상+퍼블리시티+저작권) |
| AI 생성 lookalike | Dropshot/나노바나나, ImageFX | 초상권+퍼블리시티(닮음 시) |
| 썸네일 | 자동 썸네일 생성/선택 | 상동 |

### 2.2 발동 신호
- 글 컨텍스트가 celebrity(SPEC-DEFAMATION `isCelebrityContext` 재사용) **또는** 프롬프트/키워드에 실존인물명 존재.
- 삽입 후보 이미지에 **사람 얼굴이 식별**되고, 그 얼굴이 유명인일 개연성이 있을 때.

### 2.3 범위 밖
- 얼굴 없는 이미지(제품/풍경/일러스트), 명백한 스톡/CC 라이선스.
- 실존인물이 아닌 가상 캐릭터/일반인 군중.

---

## 3. 핵심 난제 (설계 결정 필요)

### 3.1 "유명인 얼굴 식별"의 기술적 현실
- **완전 신원 매칭(이 사람이 누구다)은 과잉·고비용·프라이버시 위험** → 목표를 '유명인 여부/식별가능 얼굴 존재'로 낮춘다.
- 선택지:
  - (A) **얼굴 존재 감지만**(face detection, 온디바이스): celebrity 컨텍스트 + 얼굴 있는 사진 → 자동 삽입 보류 + 경고. 저비용·오탐 많음(일반인도 걸림).
  - (B) **얼굴 감지 + 유명인 개연성 휴리스틱**: 크롤링 출처(연예 기사/위키/인물DB) + 파일명/캡션에 인물명 → 유명인 가중. 중간 정확도.
  - (C) **Vision API 유명인 인식**(예: 클라우드 celebrity recognition): 정확하나 비용·프라이버시·약관 이슈.
- **권고: (B) 우선** — 소스 메타(연예 도메인/인물명 캡션) + 얼굴 감지 결합. celebrity 컨텍스트에서만 발동해 오탐 억제.

### 3.2 차단 vs 경고 (SPEC-DEFAMATION 결정과 정합)
- 텍스트 게이트는 '1회 확인'. 이미지는 침해가 '동의 없는 사용' 자체로 성립하고 되돌리기 어려워 **더 보수적**이어야.
- **권고: celebrity 컨텍스트에서 얼굴 식별 이미지는 자동 삽입 기본 보류(hold) + 라이선스 명확 대체 유도.** 사용자가 '직접 권리 확보' 확인 시에만 삽입.

### 3.3 대체 유도
- non-lookalike 스톡/CC0/직접 생성(실존인물 아닌) 이미지로 자동 대체 제안.
- 텍스트-이미지 일관성(QUMA-VL, `imageTextConsistencyChecker.ts`)과 충돌하지 않게.

## 4. 통합 지점 후보 (구현 시 확정)
- 이미지 수집/삽입 파이프라인(`src/image/imageFormatPipeline.ts`, 이미지 검색 수집부)에 **삽입 전 게이트** 1지점.
- celebrity 판정은 SPEC-DEFAMATION `isCelebrityContext` 재사용(중복 로직 금지).
- 얼굴 감지 라이브러리: 온디바이스(예: 번들 가능한 경량 감지기) 우선, 네트워크/비용 게이트.

---

## 5. 단계별 계획 (1릴리즈 1~3 fix)

### P0 — 진단/가시성 (릴리즈 1, 코드 변경 최소)
- celebrity 컨텍스트 + 크롤링 이미지 삽입 시 **경고 로그 + 사용자 고지**("실존인물 사진일 수 있음 — 초상권 확인"). 차단 없음. 실측으로 빈도/오탐 파악.

### P1 — 얼굴 감지 게이트 (릴리즈 1~2)
- 온디바이스 얼굴 감지 도입 → celebrity 컨텍스트 + 얼굴 있는 크롤링 이미지 자동 삽입 **보류(opt-in 삽입)**.
- 소스 메타(연예 도메인/인물명 캡션) 유명인 가중.

### P2 — AI lookalike + 대체 유도 (릴리즈 1~2)
- AI 생성 이미지의 실존인물 lookalike 억제(프롬프트 가드) + non-lookalike 대체 제안.

### 보류/별도
- 클라우드 유명인 인식(C)은 비용·프라이버시·약관 재검토 후 별도 결정.

---

## 6. 오탐 방지
- celebrity 컨텍스트 밖(맛집/IT/일반)에서는 게이트 미발동.
- 얼굴 없는 이미지·명시적 CC/스톡 라이선스는 통과.
- 경고 우선(P0) → 실측 후 보류/차단 단계 상향.

---

## 7. 미결정 / 사용자·자문 필요
1. **얼굴 감지 라이브러리**: 온디바이스 경량 vs 클라우드 정확도 — 비용/프라이버시/번들 크기 트레이드오프.
2. **차단 강도**: 보류(opt-in 삽입) vs 경고만 — 이미지는 되돌리기 어려워 보수 권고.
3. **AI lookalike 판정**: 생성 이미지가 실존인물을 닮았는지 자동 판정은 난제 → 프롬프트 억제 우선.
4. **법률 자문**: (타)목의 AI 생성 이미지 적용, 보도/논평 예외 범위 — unsettled.
5. **성능**: 발행당 이미지 여러 장 얼굴 감지 시 저사양 PC 부하(SPEC-PERF-2026 정합 필요).

---

**핵심 요약**: 초상권/퍼블리시티권은 '거짓'을 요건으로 하지 않아 텍스트 가드로 커버 불가. celebrity 컨텍스트에서 얼굴 식별 이미지 자동 삽입을 P0 경고 → P1 보류로 단계 상향하고, non-lookalike 대체를 유도한다. 얼굴 감지 방식·차단 강도·비용은 구현 착수 시 사용자 결정.
