# SPEC-CONVERSION-001 — 끝판왕 구매 전환 생성 엔진

**Status**: draft
**Created**: 2026-04-18
**Target Start**: 2026-04-29 (Wed, SPEC-REVIEW-001 P1 완료 이후)
**Target Complete**: L2 = 2026-05-12 (Mon), L3 = 2026-06-02 (Mon), L4 = 분기 단위
**Owner**: 박성현
**Depends**: SPEC-REVIEW-001 (P0~P2 완료 필수, P1은 L2-3 전제)
**Related**: SPEC-REVIEW-001 (보완 관계 — REVIEW는 안전장치, CONVERSION은 품질 극대화)
**Scope**: cross-module (`src/content`, `src/crawler`, `src/rag` 신규, `src/prompts`, `src/image`, `src/monitor`)

## Why

better-life-naver의 수익 핵심은 네이버 쇼핑커넥트(어필리에이트) 링크를 통한 **구매 전환**입니다. 발행량이나 상위 노출만으로는 한계가 명확하고, 결국 "사람이 읽고 구매 결정을 내리게 만드는 콘텐츠 품질"이 매출을 좌우합니다.

그러나 현재 생성 파이프라인의 근본 한계는 다음과 같습니다.

1. **단일 LLM 콜로 모든 것을 처리**: `src/contentGenerator.ts`(10,144줄)는 하나의 거대한 프롬프트에 카테고리 판단, 페르소나, 본문, 후킹, 팩트 정리, 전환 유도를 한 번에 요구합니다. 사람 편집자처럼 단계별 퇴고가 불가능합니다.
2. **경쟁 제품/경쟁 블로거 맥락 부재**: 가격 포지셔닝("같은 카테고리 평균 X원인데 이 제품은 Y원") 같은 전환 핵심 문장을 생성할 근거 데이터가 없습니다.
3. **성공 사례 학습 경로 없음**: 상위 노출 블로거의 실제 문구/구조가 생성 프롬프트에 전혀 반영되지 않습니다. 즉, 우리 생성기는 "잘 팔리는 글"을 본 적이 없습니다.
4. **실시간 트렌드/시즌 둔감**: 네이버 검색어 트렌드나 시즌 키워드가 본문에 자동 주입되지 않습니다.
5. **전환 결과 피드백 루프 없음**: 어떤 문구가 실제로 클릭/구매를 만들었는지 측정되지 않아 프롬프트가 개선되지 않습니다.

SPEC-REVIEW-001이 **환각을 막는 안전장치**(데이터 없으면 생성 차단)라면, 이 SPEC은 **품질을 극대화하는 가속기**(데이터가 있을 때 경쟁 블로거 대비 명확히 우월한 글을 만든다)입니다. 두 SPEC은 상호 보완 관계입니다 — REVIEW 없이 CONVERSION만 도입하면 화려한 환각 글이 쏟아지고, CONVERSION 없이 REVIEW만 있으면 안전하지만 평범한 글에 머뭅니다.

## 전략 구조 (3-레벨)

| Level | 기간 | 목표 | 범위 |
|---|---|---|---|
| L2 | 1~2주 | 체인드 LLM 파이프라인 + 경쟁 데이터 + 상위 50개 벤치마크 | 이 SPEC의 즉시 가동 범위 |
| L3 | 3~4주 | RAG 기반 성공글 참조 + 트렌드/시즌 반영 + 이미지 일관성 강화 | 벡터DB 신규 도입 |
| L4 | 분기~연간 | 편집자 LLM 퇴고 레이어 + 전환 데이터 RLHF(프롬프트 튜닝 한정) + 브랜드 톤 학습 | 실운영 데이터 축적 필수 |

L1(데이터 조건부 가드, 사회적 증거, 분량 탄력화)은 전부 **SPEC-REVIEW-001 P0~P2에 흡수**되었으므로 이 SPEC의 직접 대상이 아닙니다. 본 SPEC은 REVIEW-001이 확보한 안전한 데이터 기반 위에서만 작동합니다.

## Goals

- **G1 (L2)**: 카테고리 분류 → 페르소나 설정 → 본문 초안 → 팩트 체크 → 전환 최적화 퇴고의 **5단계 체인드 파이프라인**을 제공한다.
- **G2 (L2)**: 네이버쇼핑 검색 결과에서 유사 제품 3~5개 스펙/가격을 자동 수집하여 **가격 포지셔닝 문장** 생성 근거로 사용한다.
- **G3 (L2)**: 네이버 쇼핑 카테고리별 **상위 노출 블로그 글 50개 벤치마크**를 수집하고 구조/분량/후킹 패턴을 분석한다.
- **G4 (L3)**: 벤치마크된 성공글을 벡터DB에 저장하고 **RAG로 유사 사례 3~5개를 retrieval**하여 체인드 파이프라인에 주입한다.
- **G5 (L3)**: 네이버 검색어 트렌드/시즌 키워드를 수집하여 **본문 초안 단계에 주입**한다.
- **G6 (L4)**: 본문 초안 완성 후 "편집자 페르소나"로 **재평가·재작성하는 퇴고 레이어**를 도입한다.
- **G7 (L4)**: 발행글의 클릭률·전환율을 추적하고 성과 좋은 글의 패턴을 **프롬프트에 역반영**한다(모델 파인튜닝은 제외, 프롬프트 튜닝만).

## Non-Goals

- **NG1**: 환각 방지의 기본 방어는 본 SPEC이 다루지 않는다 — 전부 SPEC-REVIEW-001 책임.
- **NG2**: 자체 LLM 파인튜닝/재학습은 제외. L4-2는 프롬프트 튜닝(역반영) 한정.
- **NG3**: 다국어(영문/일문 등) 생성은 범위 밖.
- **NG4**: 카페 모드(SPEC-CAFE-*)는 범위 밖.
- **NG5**: 이미지 생성 엔진 자체 교체는 범위 밖 — 기존 `src/image/` 파이프라인을 그대로 사용하고 프롬프트 연결만 강화(L3-3).
- **NG6**: 경쟁 블로거의 본문을 **재생성/복제**하지 않는다. 분석(구조·분량·패턴)만 수행.

## Success Criteria

- **S1 (L2)**: 체인드 파이프라인이 feature flag `CHAINED_GEN_V1` ON 시 동작하며, 동일 입력으로 기존 단일 콜 결과 대비 **A/B 구매 전환율 +15% 이상**(1개월 발행 데이터 기준, 신뢰구간 95%).
- **S2 (L2)**: 경쟁 제품 데이터 수집기가 네이버쇼핑 검색 결과에서 유사 제품 **최소 3개**의 가격/주요 스펙을 획득하는 성공률 **80% 이상**.
- **S3 (L2)**: 상위 블로거 글 **50건 벤치마크 데이터셋**이 확보되고, 구조 패턴(섹션 수/분량/이미지 배치/CTA 위치) 통계가 산출된다.
- **S4 (L3)**: RAG retrieval이 현재 제품과 실제로 유사한 카테고리의 성공글을 **top-5 정확도 80% 이상**으로 반환한다(수작업 레이블 100건 기준).
- **S5 (L3)**: 시즌·트렌드 키워드가 본문 초안에 반영되는 단위 테스트 5건 전부 통과.
- **S6 (L4)**: 편집자 퇴고 레이어 적용 전/후 체류시간 **+20% 이상**, 스크롤 깊이 **+15% 이상**.
- **S7 (공통)**: `npm run build` 통과, 기존 236+ 유닛 테스트 회귀 **0건**, 월간 LLM API 비용이 설정한 상한(예: 기존 대비 4배) 이내.

## Constraints

- **C1**: 파일당 300줄 하드 리밋(`.claude/rules/autopus/file-size-limit.md`) 준수. 신규 파일은 전부 250줄 이하 목표.
- **C2**: 체인드 파이프라인은 **SPEC-REVIEW-001 P0 가드 통과 이후**에만 호출된다. 데이터 없을 때는 REVIEW-001의 축약 경로로 fallback.
- **C3**: 벡터DB는 로컬 우선(pgvector on 기존 MongoDB 컨테이너 대안 혹은 ChromaDB 로컬). SaaS(Pinecone 등)는 L3 내에서 옵션으로만 평가.
- **C4**: 경쟁 블로거 벤치마크 수집은 **분석 목적만**. 원문 저장은 최소화(해시·통계·구조만 영구 보관, 원문은 캐시 TTL 7일).
- **C5**: 모든 신규 LLM 콜은 `src/monitor/operationsDashboard.ts`에 단계별 토큰/비용 메트릭을 남긴다.
- **C6**: Feature flag 체계(L2 `CHAINED_GEN_V1`, L3 `RAG_RETRIEVAL_V1`/`TREND_INJECT_V1`, L4 `EDITOR_LAYER_V1`/`CONVERSION_RLHF_V1`)로 단계적 롤아웃.
- **C7**: 언어 정책: 코드 주석 영어, 커밋 메시지/응답 한국어.

## Risks

| 리스크 | 영향 | 완화 |
|---|---|---|
| R1. 체인드 파이프라인으로 LLM 비용 3~5배 증가 | 월 운영비 급등 | 단계별 캐싱(카테고리/페르소나는 제품당 1회), 상한 설정, 모델 등급 분리(초안=저렴, 퇴고=고급) |
| R2. 경쟁 블로거 크롤링의 법적 회색지대 | 저작권 분쟁 | 원문 영구 저장 금지, 구조·통계만 보관, robots.txt 준수, 수집 빈도 제한 |
| R3. LLM 제공자 변경 시 각 단계 프롬프트 재튜닝 필요 | L2/L3 재작업 | 각 단계 프롬프트를 파일로 분리(`src/prompts/affiliate/chain/`), 제공자 어댑터 계층 유지 |
| R4. RAG retrieval 품질이 낮아 노이즈 주입 | 품질 오히려 하락 | S4 수락 기준(top-5 정확도 80%)을 게이트로 사용, 실패 시 flag OFF |
| R5. 벤치마크 데이터 소수 카테고리 편중 | 특정 카테고리 품질만 향상 | 카테고리별 최소 수집 건수(각 10건) 강제 |
| R6. 편집자 레이어가 원본 의도를 훼손 | 품질 불안정 | 퇴고 전/후 diff를 로깅, 임계 이상 변경 시 원본 유지 |
| R7. 전환율 A/B 테스트 통계적 유의성 확보 어려움 | 효과 측정 불가 | 발행량 확보 전까지 체류시간/스크롤 같은 proxy 지표 병행 |
| R8. 리뷰 0건 제품은 어떤 수준도 불가 | 물리적 한계 | 이 SPEC으로 극복 불가. UI에서 "리뷰 부족 — 저품질 예상" 경고 표시(SPEC-REVIEW-001과 연계) |
