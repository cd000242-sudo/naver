# SPEC-CONVERSION-001 - Acceptance Criteria

## 기본 원칙

- 각 Level(L2/L3/L4)은 **독립적으로 수락**될 수 있다. 즉 L2만 완료된 상태로도 프로덕션 배포 가능.
- 모든 완료 기준은 **수치 증거** 또는 **파일 존재·테스트 통과**로 검증되어야 한다("잘 돌아보였다"는 수락 불가).
- 회귀: 기존 236+ 유닛 테스트 전부 통과가 모든 Level의 전제.

## L2 완료 기준 (1~2주)

### A1. 체인드 파이프라인 동작 증거

- **A1.1**: `src/content/chainedGeneration.ts`가 존재하고 `npm run build` 통과.
- **A1.2**: Feature flag `CHAINED_GEN_V1`=ON으로 동일 입력에 대해 5단계(카테고리/페르소나/초안/팩트/퇴고) 모두 호출되는 로그가 `operationsDashboard`에 기록됨.
- **A1.3**: `tests/content/chained-gen.test.ts` 3건 모두 통과(카테고리 입력 고정 시 결과 고정, 페르소나 캐시 히트, fallback 경로).
- **A1.4**: `CHAINED_GEN_V1`=OFF 시 기존 단일 콜 경로(`contentGenerator.ts`)가 변경 없이 그대로 호출됨(회귀 테스트 1건).

### A2. 경쟁 제품 수집 성공률

- **A2.1**: 샘플 제품 20개(카테고리 5종 × 4개)에 대해 유사 제품 3개 이상 확보율 **80% 이상**.
- **A2.2**: 셀렉터 엔트리 `shoppingCompetitorSelectors.ts`가 중앙 레지스트리에 등록되고 원격 패치 가능.
- **A2.3**: 수집 실패 시 가격 포지셔닝 블록이 본문에서 **완전히 생략**됨을 단위 테스트로 검증.

### A3. 벤치마크 데이터셋 확보

- **A3.1**: `data/benchmarks/` 하위에 카테고리 10종 × 최소 5건 이상, **총 50건 이상** 분석 데이터가 JSON으로 저장.
- **A3.2**: 각 항목에 구조 패턴(섹션 수, 분량, 이미지 수, CTA 위치) 통계 필드 존재.
- **A3.3**: 원문 필드는 `expires_at` TTL이 7일 이내로 설정됨을 확인.

### A4. 전환 품질 A/B (proxy 지표)

- **A4.1**: `CHAINED_GEN_V1` ON/OFF 각 100건 발행 후 체류시간 중앙값 비교. ON이 OFF 대비 **+10% 이상** 또는 동등(유의한 퇴보 없음).
- **A4.2**: 동일 기간 스크롤 깊이 70% 이상 도달률이 ON에서 OFF 대비 **동등 또는 상승**.
- **A4.3**: 구매 전환율(쇼핑커넥트 클릭→승인 데이터)이 수집 가능한 경우, **+15% 이상**이 확인되면 S1 초과 달성으로 기록. 수집 불가 시 체류시간/스크롤만 기준.

### A5. 비용 통제

- **A5.1**: L2 ON 상태에서 월간 LLM 비용이 기존 대비 **6배 이내**(경고 임계 4배, 초과 시 `operationsDashboard`에 알림).
- **A5.2**: 카테고리/페르소나 캐시 히트율 **40% 이상**.

## L3 완료 기준 (3~4주)

### B1. RAG Retrieval 정확도

- **B1.1**: 수작업 레이블 100건 기준 top-5 정확도 **80% 이상** (정확도 = 실제 관련 카테고리 성공글이 top-5에 포함된 비율).
- **B1.2**: `scripts/rag/eval-accuracy.ts`가 정확도 리포트를 자동 생성하고 CI에서 실행 가능.
- **B1.3**: Retrieval 검색 응답 시간 **1초 이하** (로컬 벡터DB 기준).

### B2. 트렌드·시즌 반영

- **B2.1**: 단위 테스트 5건 전부 통과(봄/여름/가을/겨울 시즌 분기, 급상승 키워드 주입 형식, 캐시 히트, API 실패 fallback, 중복 키워드 dedup).
- **B2.2**: Feature flag `TREND_INJECT_V1`=ON 시 본문 초안에 시즌 또는 트렌드 키워드가 최소 1개 포함되는 샘플 10건 중 **9건 이상**.

### B3. 이미지-텍스트 일관성

- **B3.1**: 샘플 20건에서 생성 이미지의 설명이 본문 주장과 일치한다고 판정되는 비율 **85% 이상**.
- **B3.2**: 불일치 감지 시 재생성 또는 교체 로직이 트리거되는 통합 테스트 3건 통과.

### B4. L3 통합 회귀

- **B4.1**: L3 3개 flag(RAG/TREND/IMG) 모두 ON 상태에서 `npm run build` + `npx vitest run` 전부 통과.
- **B4.2**: L3 OFF → ON 전환 시 기존 L2 결과와 **품질 퇴보 없음**(A4.1 기준 재측정).

## L4 완료 기준 (분기~연간)

### C1. 편집자 LLM 퇴고 레이어

- **C1.1**: `src/content/editorLayer.ts` 구현 완료, `EDITOR_LAYER_V1`=ON 시 호출 로그 확인.
- **C1.2**: 퇴고 전/후 diff 로깅 시스템 동작(`editorDiff.ts`), 변경률이 임계(예: 40%) 초과 시 원본 유지 분기 통합 테스트 통과.
- **C1.3**: A/B 샘플 각 100건 기준 체류시간 **+20% 이상**, 스크롤 깊이 **+15% 이상**(ON vs L3-only 비교).

### C2. 전환 데이터 RLHF (프롬프트 튜닝 한정)

- **C2.1**: 클릭·전환 추적 수단이 결정되고 실제로 발행 글에 부착됨(네이버 애널리틱스 연동 또는 자체 UTM pixel).
- **C2.2**: 최소 3개월(또는 발행 글 500건 이상) 데이터 축적 후 성과 상위 10%의 공통 패턴 리포트가 자동 생성됨.
- **C2.3**: 리포트 기반으로 프롬프트가 최소 1회 튜닝되고, 튜닝 전/후 전환율이 유의하게 개선됨(신뢰구간 95%).

### C3. 브랜드별 톤 학습

- **C3.1**: 다계정 환경에서 계정 A의 과거 글 10건 학습 후 생성된 새 글이 계정 A의 톤 지표(문장 길이 평균, 이모지 빈도, 호칭)와 **80% 이상 유사**.
- **C3.2**: 계정 전환 시 페르소나 자동 갱신 동작 확인(통합 테스트 1건).

## 공통 비기능 기준

### N1. 빌드·테스트 회귀

- **N1.1**: `npm run build` 통과 (exit 0, 0 errors).
- **N1.2**: `npx vitest run` 기존 236+ 유닛 테스트 **회귀 0건**.
- **N1.3**: 신규 추가 테스트 최소 수:
  - L2: +15건
  - L3: +10건
  - L4: +7건

### N2. 파일 크기

- **N2.1**: 신규 TS 파일 모두 **300줄 이하**(`.claude/rules/autopus/file-size-limit.md` 준수).
- **N2.2**: 기존 `contentGenerator.ts`(10,144줄)는 본 SPEC에서 분할하지 않지만, 체인드 파이프라인이 해당 파일에 **새 코드를 추가하지 않고** 어댑터만 추가하는 것이 원칙.

### N3. Feature Flag 통제

- **N3.1**: 9개 flag 전부 환경변수 또는 설정 파일로 ON/OFF 가능.
- **N3.2**: 모든 flag 기본값은 OFF. 단계적 롤아웃 후 ON 전환.

### N4. 비용 관측

- **N4.1**: `operationsDashboard`에 단계별 토큰·비용 메트릭이 노출됨.
- **N4.2**: 월간 LLM 비용 상한이 설정 가능하고, 80% 도달 시 경고, 100% 도달 시 자동 fallback(단일 콜 경로).

### N5. 저작권·법적 리스크

- **N5.1**: 벤치마크 원문이 영구 저장되지 않음을 코드 리뷰로 확인.
- **N5.2**: robots.txt 준수, 수집 빈도 제한(카테고리당 하루 1회 이하) 구현.

## 검증 시나리오 (Gherkin-ish)

### Scenario-1: L2 체인드 정상 흐름

```
Given SPEC-REVIEW-001 P0~P2가 완료되어 있고
And CHAINED_GEN_V1=ON, COMPETITOR_DATA_V1=ON, BENCHMARK_SEED_V1=ON
When 쇼핑 카테고리 "키보드" 제품 URL로 발행 요청
Then 5단계 호출 로그가 전부 기록되고
And 본문에 "같은 카테고리 평균 X원" 형식의 가격 포지셔닝 문장이 포함되며
And 금지 경험 표현(REVIEW-001의 reviewGuard 기준)이 0개이고
And targetLength이 리뷰 개수에 연동된 값으로 결정됨
```

### Scenario-2: L2 데이터 부족 Fallback

```
Given reviews=[], 경쟁 제품 수집 실패, 벤치마크 미조회
When 발행 요청
Then REVIEW-001 P0 축약 경로가 호출되고
And 본문 분량이 1,200자 이하로 제한되며
And "리뷰 부족" 경고가 UI에 표시됨
```

### Scenario-3: L3 RAG 주입

```
Given RAG_RETRIEVAL_V1=ON, 벡터DB에 벤치마크 50건 적재 완료
When "무선 이어폰" 제품 URL로 발행 요청
Then retriever가 top-5 유사 성공글을 반환하고
And Stage 3 프롬프트에 구조 패턴 힌트가 주입되며
And 생성된 본문 구조가 top-5의 평균 섹션 수와 유사(±2 섹션)
```

### Scenario-4: L4 편집자 과도 변경 방지

```
Given EDITOR_LAYER_V1=ON, 변경률 임계 40%
When 편집자 레이어가 본문을 60% 재작성
Then diff 로깅에 변경률 60% 기록되고
And 원본이 유지되며 편집 결과는 폐기됨
And operationsDashboard에 "편집 거부" 카운터 증가
```
