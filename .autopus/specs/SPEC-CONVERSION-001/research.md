# SPEC-CONVERSION-001 - Research

## 현재 한계 요약 (코드 기반 사실)

### R1. 단일 LLM 콜 구조

- `src/contentGenerator.ts` 총 **10,144줄**. 모든 생성 로직(카테고리 판단, 페르소나, 초안, 팩트 정리, 전환 문구)이 단일 함수 경로에 집중되어 있음.
- 단일 콜로 인해 사람 편집자처럼 "초안 작성 → 읽고 다시 고치기"의 반복적 퇴고가 불가.
- 모든 변경이 동일 프롬프트에서 경쟁하기 때문에 한 관심사(예: 팩트) 강조 시 다른 관심사(예: 후킹)가 약해지는 **트레이드오프가 명시적으로 보이지 않음**.

### R2. 벤치마크 데이터 부재

- 현재 소스 트리에 상위 노출 블로그 글의 **구조·패턴 분석 데이터가 전혀 없음**. (`data/benchmarks/` 디렉토리 자체가 미존재)
- `src/contentGenerator.ts` 내 프롬프트는 경험칙·추측 기반 — "잘 팔리는 글의 실제 특징"이 근거로 반영되지 않음.

### R3. 경쟁 제품 맥락 부재

- `src/crawler/`는 대상 제품의 스펙/리뷰는 수집하지만(`productSpecCrawler.ts`, `shopping/`), **같은 카테고리 경쟁 제품은 수집하지 않음**.
- 결과적으로 "이 제품이 왜 좋은가"가 절대적 장점만 기술되고 **상대적 가치(가격 포지셔닝, 스펙 비교)** 문장이 생성 근거 없음.

### R4. 리뷰 의존도

- 쇼핑 모드는 `src/sourceAssembler.ts:3763-3795`에서 최대 5개 리뷰 수집. 리뷰 0건 제품은 어떤 Level의 엔진으로도 **"리뷰 풍부한 글" 수준의 구매 전환율 달성 불가**(물리적 한계).
- 본 SPEC은 리뷰 있는 제품의 품질을 극대화하는 것이 목표. 리뷰 0건 제품은 SPEC-REVIEW-001의 P0 축약 경로로 안전하게 처리하되 UI에서 "저품질 예상" 경고를 표시.

### R5. 전환 피드백 루프 부재

- `src/monitor/operationsDashboard.ts`는 셀렉터·품질·발행·세션 지표는 있으나 **클릭·구매 전환 지표는 없음**.
- 어떤 문구/구조가 실제 구매를 만들었는지 측정 수단 자체가 없어 프롬프트 개선이 경험칙에만 의존.

### R6. 이미지-텍스트 정합성의 제한적 범위

- `src/image/imageTextConsistencyChecker.ts`는 존재하지만 주로 **이미지-본문 불일치 방지**에 초점. 본문 주장을 이미지가 **적극적으로 뒷받침**하는 방향은 미구현.

## 설계 결정 근거

### D1. 체인드 vs 단일 LLM 콜

**결정**: 5단계 체인드 채택(L2-1).

| 기준 | 단일 콜 | 체인드 |
|---|---|---|
| 비용 | 1x | 3~5x |
| 지연 | 1x (병렬 불가) | 3~5x (순차) |
| 각 관심사 품질 | 상호 경쟁 | 단계별 전담 |
| 디버깅 가능성 | 낮음(블랙박스) | 높음(단계별 로그) |
| 사람 편집자 시뮬레이션 | 불가 | 가능 |

체인드의 비용·지연 증가는 (a) 카테고리·페르소나 캐시, (b) 단계별 모델 등급 분리(초안=저렴 모델, 퇴고=고급 모델)로 완화. 품질 향상의 상한이 크기 때문에 경쟁 블로거 대비 우월 달성이 목표인 본 SPEC과 정합.

### D2. RAG vs 퓨샷 프롬프트

**결정**: RAG 채택(L3-1). 단, 초기에는 하이브리드(top-5 retrieval 결과를 퓨샷 형태로 삽입).

| 기준 | 퓨샷(정적 예시) | RAG |
|---|---|---|
| 현재 제품 유사성 | 낮음(모든 제품에 동일 예시) | 높음(카테고리/특징별 retrieval) |
| 벤치마크 확장성 | 수동 업데이트 | 자동 |
| 구현 비용 | 낮음 | 중간(벡터DB 도입) |
| 품질 상한 | 낮음 | 높음 |

L2 단계에서는 퓨샷으로 시작(`BENCHMARK_SEED_V1`), L3에서 RAG로 전환하면서 비용 대비 품질 효과를 측정.

### D3. 벡터DB 선택 (예비 결정, L3-1에서 확정)

| 옵션 | 장점 | 단점 | 권장도 |
|---|---|---|---|
| ChromaDB 로컬 | 설치 간단, SQLite 기반 | 대규모 스케일 취약 | **초기 권장** |
| pgvector on MongoDB 대체 PG | 관계형과 결합 용이 | 현재 스택이 Mongoose — PG 추가 비용 | 중간 |
| Pinecone(SaaS) | 스케일·관리 편의 | 월 비용, 데이터 반출 | 장기만 |
| Qdrant 로컬 | 성능 우수 | 도입 복잡도 | 중간 |

**예비 결정**: ChromaDB 로컬(50~500건 수준에서 충분, Electron 앱 내 번들링 가능성 검토). 최종은 L3-1 기술 결정 문서에서 확정.

### D4. 임베딩 모델

| 옵션 | 한글 품질 | 비용 | 레이턴시 |
|---|---|---|---|
| OpenAI text-embedding-3-small | 높음 | API 콜 | 100~300ms |
| OpenAI text-embedding-3-large | 매우 높음 | 3x 비싼 | 200~500ms |
| 로컬 sentence-transformers (ko-sroberta) | 중간 | 0 | <50ms |
| Gemini embedding-001 | 높음 | API 콜 | 100~300ms |

**예비 결정**: L3에서는 OpenAI text-embedding-3-small 우선 도입(한글 품질 우선). 임베딩은 문서당 1회 영구 캐시이므로 비용 영향 제한적.

### D5. 오픈소스 vs SaaS 벡터DB

- 오픈소스(ChromaDB/Qdrant 로컬): **데이터 주권 유지**, Electron 앱 내 번들링 시 사용자 머신에서 동작 가능. 벤치마크가 회색지대 저작권 이슈를 안고 있으므로 외부 전송 최소화가 바람직.
- SaaS: L4에서 브랜드 톤 학습·다계정 운영 시 중앙 집중화 필요하면 재검토.

### D6. 경쟁 블로거 글 "분석만, 재생성 금지" 원칙

- 구조(섹션 수/분량/이미지 수/CTA 위치), 키워드 빈도, 후킹 패턴은 **통계**이므로 분석 산출물.
- 원문 문장을 그대로 재사용하거나 재구성하지 않는다. 원문은 7일 캐시 후 삭제.
- 분석 결과를 "구조 힌트"로만 프롬프트에 주입(예: "이 카테고리 성공글의 평균 섹션 수는 6개, 평균 분량 2,000자"). 문장 자체는 주입하지 않음.

## 경쟁 벤치마크 — 네이버 쇼핑 상위 블로거 글 조사 항목

L2-4에서 수집할 분석 항목(카테고리당 최소 5건):

1. **구조 패턴**: 섹션 수, 섹션 제목 패턴(의문형/선언형 비율), 이미지-텍스트 블록 교차 간격
2. **분량**: 총 글자 수, 섹션별 평균 분량, 상위/하위 25% 편차
3. **후킹 유형**: 도입부 첫 문장 패턴(질문/통계/경험 선언/가격 강조)
4. **사회적 증거 사용**: "~명이 구매", "후기 ~개" 등 수치 언급 빈도·위치
5. **CTA 위치·유형**: 쇼핑커넥트 링크의 본문 내 위치(상단/중단/하단), 직접 구매 유도 문구 유형
6. **이미지 구성**: 제품 사진/상세 컷/비교표/텍스트 오버레이 비율
7. **톤·어휘**: 존대/반말, 이모지 빈도, 문장 평균 길이

이 7개 항목을 `benchmarkAnalyzer.ts`가 JSON으로 산출하고 RAG 벡터 문서의 metadata로 저장.

## 참고 기법

### T1. Chain-of-Thought (CoT)

- 각 체인드 단계 내부에서도 "먼저 X 생각 → 그 다음 Y 결정"의 명시적 reasoning을 프롬프트에 요구하여 품질 향상.
- 단, 최종 출력에는 reasoning을 제거(소비자용 본문에는 불필요).

### T2. Retrieval-Augmented Generation (RAG)

- L3-1의 핵심. 벤치마크 50건을 벡터화 → 현재 제품과 유사한 top-5 retrieval → 프롬프트에 구조 힌트 주입.
- 유사도 기준: 제품 카테고리(1차) + 가격대(2차) + 키워드 중복(3차).

### T3. Constitutional AI (CAI)

- L4-1 편집자 레이어에 적용. "독자 입장에서 부자연스럽거나 과장된 문장을 찾아 고쳐라" 같은 메타 규칙으로 퇴고.
- 단, 과도한 재작성은 원본 의도 훼손 가능 — diff 기반 변경률 임계(예: 40%) 이상 시 원본 유지.

### T4. RLHF with Outcome Signals (프롬프트 튜닝 한정)

- L4-2에서 실 구매 전환 데이터를 활용. 단, 모델 파인튜닝은 범위 밖.
- 상위 성과 글의 공통 패턴을 **사람이 해석 가능한 리포트**로 생성 → 프롬프트에 반영.
- 예: "상위 10% 글은 도입부에 가격 강조가 있다" → 체인드 Stage 3 프롬프트에 가격 강조 지침 추가.

### T5. Few-Shot with Similarity-Weighted Retrieval

- L2 초기에 벤치마크를 퓨샷으로 사용할 때 정적 나열보다 **카테고리별 가중치**로 선택.

## 사회적 증거·가격 포지셔닝 문장 패턴

벤치마크 분석 후 확정되는 패턴이지만, 예상 템플릿:

- "같은 ~카테고리 평균 가격은 X원대인데, 이 제품은 Y원으로 ~% 저렴합니다"
- "네이버에서 ~개 구매되었고 후기 ~개가 달렸어요"
- "같은 가격대 제품 중 ~기능이 있는 건 이 제품뿐"

이들은 `src/content/pricePositioning.ts`(L2-2 산출물)가 경쟁 데이터로부터 자동 생성.

## 한계 인정 (Honest Limits)

### H1. 리뷰 0건 제품은 도달 불가능

- 어떤 Level의 엔진도 리뷰 없는 제품을 "리뷰 풍부한 글" 수준으로 끌어올릴 수 없음.
- 본 SPEC은 이 한계를 인정하고, 리뷰 부족 시 SPEC-REVIEW-001의 P0 축약 경로로 **안전하게 저품질을 선언**하는 쪽을 선택한다. 억지로 내용을 부풀리는 것은 환각 위험이 더 크다.

### H2. LLM API 비용 증가

- L2 체인드만 해도 기존 대비 3~5배, L4 편집자까지 켜면 4~7배.
- 완화: 단계별 모델 등급 분리, 캐싱, 월간 비용 상한과 자동 fallback.
- 그러나 **품질/비용 트레이드오프**가 본질적 존재. "체인드 OFF가 여전히 동작해야 함" 원칙은 그래서 hard 요건.

### H3. 벤치마크의 법적 리스크

- 경쟁 블로거의 공개 글을 자동 수집하여 분석하는 것은 한국 저작권법상 **회색지대**.
- 리스크 완화: (a) 분석·통계만 저장, (b) 원문 7일 TTL, (c) 재생성 금지 원칙, (d) robots.txt 준수, (e) 수집 빈도 제한.
- 완전히 제거하는 유일한 방법은 공식 데이터셋 구매 또는 라이선스 확보인데 한국어 블로그 데이터는 상업적 라이선스가 희박. 현재 리스크 완화 전략을 채택.

### H4. LLM 제공자 변경 시 재튜닝 필요

- Claude/Gemini/GPT는 동일 프롬프트에도 다른 반응. 체인드 단계별 프롬프트를 파일로 분리한 이유.
- 제공자 어댑터는 이미 프로젝트에 존재하지만 **프롬프트 자체는 제공자별로 재튜닝이 필요**할 수 있음. 각 단계 프롬프트에 제공자별 variant를 두는 것이 장기 운영에 유리.

### H5. A/B 테스트의 통계적 유의성

- 하루 발행량이 제한적인 블로그 자동화에서 통계적 유의성(p<0.05) 확보까지 수백~수천 건 발행이 필요.
- 초기에는 체류시간/스크롤 같은 proxy 지표로 방향성 판단, 누적 데이터 500건 이상 시 구매 전환까지 확인.

## Open Questions

### Q1. 벤치마크 수집 방식

- (a) 네이버 공식 검색 API — 정확하지만 블로그 글 본문 접근 제한
- (b) 브라우저 크롤링(기존 `src/crawler/` 인프라 활용) — 가장 현실적이나 저작권 리스크
- (c) 수동 URL 목록 — 50건은 사람이 수집 가능하나 확장성 없음
- **예비 답**: (b) 브라우저 크롤링 + 원문 7일 TTL + 분석만 영구 저장. L2-4 기술 결정 시점에서 재확정.

### Q2. 벡터DB 최종 선택

- 현재 예비 결정은 ChromaDB 로컬. 하지만 Electron 앱 내 번들링 가능성, 사용자 머신의 디스크 사용량, 다계정 시 격리 전략이 L3-1 착수 전 검증 필요.

### Q3. 임베딩 제공자

- OpenAI vs Gemini vs 로컬. 한글 벤치마크 10건으로 top-5 정확도 비교 실험 필요. 차이가 5% 이내면 비용 우선(로컬 모델).

### Q4. 체인드 단계별 LLM 등급 분리

- 예비안: 초안(Gemini Flash/GPT-4o mini) → 팩트(Claude Sonnet) → 퇴고(Claude Opus).
- 비용 시뮬레이션 필요. 초기에는 모든 단계 동일 모델(현재 기본 모델)로 시작 후, 분리의 품질 영향 측정.

### Q5. 전환 데이터 수집 수단

- (a) 네이버 애널리틱스 API — 공식이나 제한적 메트릭
- (b) 자체 UTM pixel on 쇼핑커넥트 리다이렉트 — 정밀하나 구현 비용
- (c) 쇼핑커넥트 대시보드의 승인 데이터 CSV 주기적 동기화 — 가장 확실한 수익 신호
- L4-2 착수 시 결정. 가능하면 (a)+(c) 병행.

### Q6. 편집자 레이어 재작성 허용 한도

- 변경률 임계(예: 40%) 결정에 실측이 필요. 10~50% 범위에서 A/B 실험 후 확정.
- 변경률 산정 방식: 문자 단위 Levenshtein vs 토큰 단위 diff. 후자가 의미 단위에 더 근접할 가능성.

### Q7. 벤치마크 갱신 주기

## Summary

이 SPEC은 SPEC-REVIEW-001이 확보한 "안전한 데이터 기반" 위에서, 체인드 LLM·벤치마크·RAG·편집자 퇴고를 순차 도입하여 **경쟁 블로거 대비 명확히 우월한 구매 전환율**을 목표로 한다.

핵심 인사이트는 두 가지:

1. **품질은 단일 콜의 프롬프트 튜닝으로 달성 불가** — 사람 편집자처럼 단계별로 끊어서 각자의 책임을 지워야 한다.
2. **벤치마크 없이는 우월을 정의할 수 없다** — "이 카테고리에서 실제로 잘 되는 글"이 무엇인지 데이터로 알아야 목표를 설정할 수 있다.

L2(1~2주)로 체인드+벤치마크 기반을 깔고, L3(3~4주)로 RAG+트렌드로 품질 상한을 끌어올리고, L4(분기~연간)로 편집자 레이어·전환 데이터 루프를 가동한다. 각 Level은 독립적으로 수락·배포 가능하며, 실패 시 feature flag로 즉시 롤백.
